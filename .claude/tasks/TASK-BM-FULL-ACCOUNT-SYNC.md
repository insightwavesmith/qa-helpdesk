# TASK: 비즈니스 매니저 전체 계정 데이터 수집 체계 (BM Full Account Sync)

## 타입
기획

## 이게 뭔지
비즈니스 매니저에 등록된 150개 광고계정의 광고 콘텐츠를 전부 수집하는 데이터 인프라 설계. 현재는 수강생이 직접 등록한 45개만 수집 중.

## 왜 필요한지
- Smith님 원래 설계: **모든 계정 데이터를 먼저 수집 → 그 위에 벤치마크/분석 데이터 축적 → 수강생은 자기 계정 데이터만 꺼내 쓰는 구조**
- 현재 구현은 수강생 등록 계정만 수집 → 전체 데이터의 30%만 확보 → 벤치마크 모수 부족, 업종별 비교 불가
- 105개 계정의 광고 소재/성과 데이터가 아예 없어서 총가치각도기 벤치마크 정확도 저하

## 현황 (2026-03-30 기준)

| 항목 | 현재 | 목표 |
|------|------|------|
| 수집 계정 | 45개 (수강생 직접 등록) | 150개 (BM 전체) |
| 데이터 흐름 | 수강생 등록 → 수집 | BM 전체 수집 → 수강생은 자기 것 조회 |
| 벤치마크 모수 | 45개 계정 | 150개 계정 |

## 기존 인프라 (이미 준비된 것)
- `ad_accounts.is_member` 컬럼: 수강생/비수강생 구분 가능
- `collect-daily`: `active=true` 전체 수집, is_member 무관
- 하위 테이블(creatives, creative_media, daily_ad_insights) 전부 `account_id` FK
- ADR-001: 계정 종속 구조 확정 — 구조 변경 불필요

## PM이 결정해야 할 것

### 1. 데이터 수집 범위
- 150개 전체 매일 수집? 활성 계정만?
- ACTIVE 광고만? PAUSED/ARCHIVED 포함?
- 과거 데이터 백필 범위 (90일? 180일? 전체?)

### 2. 수강생 매칭 UX
- 현재: 수강생이 계정 ID 직접 입력해서 등록
- 변경안: BM에서 이미 수집된 계정 목록 보여주고 선택
- `is_member=true` + `user_id` 연결 시점/방식

### 3. 비수강생 데이터 활용
- 벤치마크 집계에만 사용? (익명 통계)
- 업종별/카테고리별 분류 필요?
- 수강생에게 비수강생 개별 데이터 노출 범위

### 4. 인프라 영향
- API 호출량 3배 증가 (Meta rate limit 설계)
- GCS 저장량/Cloud Run 호출량 증가 예측
- collect-daily 크론 실행 시간 증가 (현재 ~3분 → 예상 ~10분)

## 구현은 간단 (코드 수준)
아키텍처 재설계 아님. 크론 1개 추가 수준:
```
새 크론: sync-bm-accounts
  → Meta API: GET /{business_id}/owned_ad_accounts + client_ad_accounts
  → ad_accounts UPSERT (is_member=false, active=true)
  → 기존 collect-daily가 자동으로 신규 계정도 수집
```
