## 1. 브릭 = 도구다

브릭은 서비스가 아니다. **도구**다.

| 서비스 | 도구 |
|--------|------|
| 정해진 길이 있다. "이렇게 쓰세요" | 정해진 길이 없다. "뭘 만들지는 당신이 정해" |
| Notion, Jira, Linear | **Minecraft, Scratch, EVE Online** |
| 기능을 쓴다 | 세계를 만든다 |

브릭이 참고한 3개 도구:

### Minecraft — "블록으로 내 세계를 건축"
마인크래프트에서 블록은 그냥 재료다. 집을 짓든, 성을 짓든, 레드스톤 회로를 만들든 사용자가 정한다.
브릭도 마찬가지다. Plan 블록, Do 블록, Check 블록은 재료일 뿐. 어떤 워크플로우를 만들지는 사용자가 정한다.

### Scratch — "Low Floor / High Ceiling / Wide Walls"
- **Low Floor** (쉬운 시작): 프리셋 하나 클릭하면 PDCA가 바로 돌아간다
- **High Ceiling** (높은 천장): 커스텀 Gate, Loop, Compete 조합으로 복잡한 워크플로우도 가능
- **Wide Walls** (넓은 벽): 같은 결과를 내는 방법이 여러 가지 — 3축 조합의 자유도

### EVE Online — "개발사는 도구만 제공, 세계는 플레이어가 만듦"
EVE에서 CCP(개발사)는 우주와 물리 법칙만 만들었다. 경제, 전쟁, 정치, 동맹은 전부 플레이어가 만들었다.
브릭도 마찬가지다. 우리는 엔진(물리 법칙)만 만든다. 워크플로우(세계)는 사용자가 만든다.

## 2. 3축 — Brick / Team / Link

브릭의 모든 것은 3개의 독립된 축으로 돌아간다.

```
┌─────────────────────────────────────────────────────┐
│                    Brick 엔진                        │
│                                                     │
│   🧱 Brick (블록)     → "뭘 하는가"     → 제어      │
│   👥 Team (팀)        → "누가 하는가"    → 자유      │
│   🔗 Link (연결)      → "어떤 순서로"    → 자유      │
│                                                     │
│   Brick = 제어 (Gate로 품질 강제)                     │
│   Team, Link = 자유 (사용자가 마음대로 조합)           │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 왜 3축인가?

어떤 작업이든 3가지 질문으로 분해된다:
1. **뭘 하는가?** → 기획? 설계? 구현? 테스트? (Brick)
2. **누가 하는가?** → PM팀? CTO팀? 외부 API? 사람? (Team)
3. **어떤 순서로?** → 순차? 동시에? 실패하면 되돌아가? (Link)

이 3가지가 **독립적**이라는 게 핵심이다.

```
같은 블록(Plan)인데:
  - PM팀이 할 수도 있고, 마케팅팀이 할 수도 있다 (Team만 바꿈)
  - 다음에 Design으로 갈 수도, 바로 Do로 갈 수도 있다 (Link만 바꿈)

같은 팀(CTO팀)인데:
  - Do 블록을 할 수도, Check 블록을 할 수도 있다 (Brick만 바꿈)
  - Claude Agent Teams로 할 수도, Codex로 할 수도 있다 (Adapter만 바꿈)

같은 연결(loop)인데:
  - check→do 루프일 수도, review→design 루프일 수도 있다 (Brick만 바꿈)
```

**한 축을 바꿔도 다른 축이 안 깨진다.** 이게 확장성의 핵심이다.

### 비유: 레고

```
레고 = Brick(블록 모양) × 색깔(Team) × 연결 방식(Link)

같은 2×4 블록을 빨간색으로 쓸 수도, 파란색으로 쓸 수도 있다.
위에 쌓을 수도, 옆으로 붙일 수도, 경첩으로 꺾을 수도 있다.
모양 × 색깔 × 연결이 독립적이라서 무한한 조합이 가능하다.
```

## 3. 축1: Brick (블록) — "뭘 하는가"

### 개념

블록은 **하나의 작업 단위**다. "기획해", "설계해", "구현해", "테스트해" — 각각이 하나의 블록.

### 블록의 구성

```
┌──────────────────────────────────┐
│           Plan 블록               │
│                                  │
│  입력: TASK 설명, 프로젝트 컨텍스트  │
│  실행: Team이 작업 수행            │
│  출력: Plan 문서 (artifacts)       │
│  검증: Gate가 품질 체크            │
│                                  │
│  Gate: "Plan 문서가 있나?"         │
│        "5페이지 이상인가?"         │
│        "필수 섹션이 다 있나?"       │
│                                  │
└──────────────────────────────────┘
```

### 할 수 있는 것

| 기능 | 설명 | 비유 |
|------|------|------|
| **블록 생성** | 새로운 작업 단위 만들기 | 마인크래프트에서 새 블록 종류 만들기 |
| **블록 편집** | 이름, 설명, Gate 규칙 변경 | 블록에 인챈트 거는 것 |
| **Gate 설정** | 블록 완료 조건 정의 (7종) | 검수대 설치 |
| **프리셋** | 블록 조합 템플릿 저장/불러오기 | 마크 건축 도면 저장 |
| **커스텀 블록타입** | 자기만의 블록 종류 정의 | 모드에서 새 블록 추가 |

### Gate 7종 (품질 검증)

블록이 끝나면 자동으로 Gate가 검사한다. Gate를 통과해야 다음으로 넘어간다.

| Gate | 비유 | 예시 |
|------|------|------|
| **command** | 기계 검수 | `pytest`가 다 통과하면 OK |
| **http** | 외부 API 확인 | 배포 URL에 200 뜨면 OK |
| **prompt** | AI 평가관 | "이 기획서가 완전한가?" AI가 판단 |
| **agent** | AI 심사위원 | 별도 에이전트가 코드 리뷰 |
| **review** | 동료 평가 | R-Brick 리뷰 → 개선 제안 |
| **metric** | 수치 검증 | 테스트 커버리지 90% 이상이면 OK |
| **approval** | 사람 승인 | CEO가 "승인" 버튼 클릭해야 다음 진행 |

**제어의 의미**: Gate가 있으니까 블록은 "제어" 축이다. 아무렇게나 통과 못 한다. 기준을 세워놓으면 엔진이 강제한다.

## 4. 축2: Team (팀) — "누가 하는가"

### 개념

팀은 **작업을 실행하는 주체**다. AI 에이전트일 수도, 사람일 수도, 외부 서비스일 수도 있다.

### 팀의 구성

```
┌──────────────────────────────────┐
│           CTO팀                   │
│                                  │
│  팀원: Claude (리더), 팀원 2명     │
│  Adapter: claude_agent_teams     │
│  MCP 도구: git, file, search     │
│  스킬: TypeScript, Python        │
│  모델: claude-opus-4-6           │
│                                  │
└──────────────────────────────────┘
```

### 할 수 있는 것

| 기능 | 설명 | 비유 |
|------|------|------|
| **팀 생성** | 새 팀 만들기 (이름, 역할) | 회사에 새 부서 만들기 |
| **팀원 추가/제거** | 팀 구성 변경 | 인사 이동 |
| **Adapter 설정** | 팀이 어떤 도구로 일할지 | "이 부서는 전화 쓸지, 슬랙 쓸지" |
| **MCP 도구** | 팀이 쓸 수 있는 도구 목록 | 부서에 비치된 장비 |
| **스킬 설정** | 팀의 전문 분야 | 부서 직무 기술서 |
| **모델 변경** | AI 에이전트의 두뇌 교체 | 인재 영입 |

### Adapter 9종 (연결 방식)

Adapter는 Team의 **속성**이다. 축이 아니다. "팀이 어떤 방식으로 연결되는가"일 뿐.

| Adapter | 누가 | 어떻게 |
|---------|------|--------|
| **claude_agent_teams** | Claude AI 팀 | tmux 세션에서 Agent Teams 모드로 작업 |
| **claude_code** | Claude AI 단독 | 단독 CLI로 작업 |
| **codex** | OpenAI Codex | ACP 하네스로 연결 |
| **human** | 사람 | 파일 마커로 완료 표시 |
| **human_management** | 관리자 | 승인/반려 UI |
| **management** | 자동 관리 | 시스템 자동 처리 |
| **mcp_bridge** | MCP 서버 | MCP 프로토콜로 연결 |
| **webhook** | 외부 서비스 | HTTP 호출로 트리거 |
| **base** | 기본 | 커스텀 어댑터의 베이스 |

**자유의 의미**: 같은 블록을 다른 팀에 배정할 수 있다. PM팀이 하던 Plan을 마케팅팀이 할 수도 있다. 팀만 바꾸면 된다.

## 5. 축3: Link (연결) — "어떤 순서로"

### 개념

Link는 **블록 간의 연결 규칙**이다. "A가 끝나면 B로" 뿐 아니라, "A가 실패하면 C로 돌아가" 같은 복잡한 흐름도 가능.

### 할 수 있는 것

| 기능 | 설명 | 비유 |
|------|------|------|
| **순차 연결** | A → B → C | 컨베이어 벨트 |
| **분기** | 조건에 따라 B 또는 C로 | 철로 스위치 |
| **루프** | 실패하면 되돌아가기 | "불합격이면 재시험" |
| **병렬** | A와 B를 동시에 | 두 라인 동시 가동 |
| **경쟁** | 먼저 끝나는 쪽 채택 | 경매 — 최고 산출물 채택 |
| **크론** | 시간 기반 반복 | 매일 오전 9시 실행 |

### Link 7종

| Link | 비유 | 예시 |
|------|------|------|
| **sequential** | 줄 서기 | plan → design → do |
| **branch** | 갈림길 | "프론트면 A팀, 백엔드면 B팀" |
| **loop** | 되돌아가기 | check 실패 → do 재실행 (최대 3회) |
| **parallel** | 동시 출발 | Plan과 Research를 동시에 |
| **compete** | 경쟁 입찰 | 2개 팀이 Design → 더 좋은 것 채택 |
| **cron** | 알람 시계 | 매일 자정에 보고서 생성 |
| **custom** | 직접 만들기 | 사용자 정의 조건부 연결 |

**자유의 의미**: 같은 블록 조합이라도 Link만 바꾸면 완전히 다른 프로세스가 된다.

```
PDCA (순차): Plan → Design → Do → Check → Act
PDCA (루프): Plan → Design → Do → Check ─실패→ Do (반복)
PDCA (경쟁): Plan → Design → [CTO-1 Do ⟷ CTO-2 Do] → Check
```

## 6. 3축의 자유도 — 왜 이렇게 만들었나

### 제어와 자유의 균형

```
┌─────────────────────────────────────────────┐
│                                             │
│   Brick (블록)  ←── 제어 ──→  Gate가 강제    │
│                                             │
│   Team (팀)     ←── 자유 ──→  자유 조합      │
│                                             │
│   Link (연결)   ←── 자유 ──→  자유 조합      │
│                                             │
└─────────────────────────────────────────────┘
```

- **Brick은 제어**: Gate가 있다. "이 블록은 반드시 이 기준을 통과해야 한다." 이건 강제다. CEO 승인 Gate를 넣으면, CEO가 승인 안 하면 절대 다음으로 못 간다.
- **Team은 자유**: 누가 하든 상관없다. Claude가 하든, Codex가 하든, 사람이 하든. 결과물이 Gate를 통과하면 된다.
- **Link는 자유**: 어떤 순서로 하든 상관없다. 순차로 하든, 병렬로 하든, 경쟁시키든. Gate만 통과하면 된다.

### ECS 패턴과의 매핑

게임 엔진에서 가장 확장성 높은 아키텍처 패턴인 Entity-Component-System과 정확히 매핑된다.

| ECS | Brick |
|-----|-------|
| Entity (존재) | Workflow Instance (워크플로우 한 판) |
| Component (데이터) | Brick + Team + Link (각각 독립 컴포넌트) |
| System (로직) | Engine (상태 머신, Gate 실행, 이벤트 버스) |

**핵심**: 상속이 아니라 **조합**. 워크플로우에 컴포넌트(Brick, Team, Link)를 붙였다 떼면 동작이 바뀐다. "레고 브릭처럼 조립"이라는 이름 자체가 이 철학이다.

### 독립성이 왜 중요한가

```
Team 축을 바꿔도 Brick과 Link는 안 깨진다:
  "Claude Agent Teams를 Codex로 교체" → Brick(블록 구조), Link(연결 순서) 변경 없음

Link 축을 바꿔도 Brick과 Team은 안 깨진다:
  "순차를 병렬로 변경" → Brick(블록 내용), Team(팀 구성) 변경 없음

Brick 축을 바꿔도 Team과 Link는 안 깨진다:
  "Check 블록에 Gate 추가" → Team(누가 하는지), Link(연결 순서) 변경 없음
```

이게 보장되니까 **나중에 한 축만 확장해도 나머지가 안 무너진다.**

## 7. 아키텍처 — 어떻게 이어져서 가능한지

### 비개발자용: 비유

```
                    ┌─────────────────┐
                    │   대시보드 (UI)   │ ← 사용자가 보는 화면
                    │  블록 배치, 연결  │
                    └────────┬────────┘
                             │ "이 워크플로우 실행해"
                    ┌────────┴────────┐
                    │   Express 서버   │ ← 접수 창구 (API)
                    │  "알겠습니다"     │
                    └────────┬────────┘
                             │ "엔진아, 이거 돌려"
                    ┌────────┴────────┐
                    │   Python 엔진    │ ← 공장 (실제 실행)
                    │  상태 머신       │
                    │  Gate 검증       │
                    │  Link 평가       │
                    │  Adapter 호출    │
                    └─────────────────┘
```

**규칙 하나**: 대시보드와 Express는 "읽기"만 한다. "쓰기"(워크플로우 시작, 블록 완료)는 반드시 Python 엔진을 거친다. 이걸 INV-EB-1이라고 부른다.

왜? Express가 직접 DB를 바꾸면 Gate/Link/Adapter가 동작하지 않는다. 엔진을 우회하면 검수 없이 공장을 통과하는 거다.

### 개발자용: 기술 스택

| 계층 | 기술 | 역할 |
|------|------|------|
| **프론트엔드** | React + React Flow + Vite | 캔버스 편집기, 실시간 상태 표시 |
| **API 서버** | Express.js (port 3200) | REST API 62개, WebSocket 실시간 이벤트 |
| **엔진** | Python FastAPI (port 3202) | 상태 머신, Gate 실행, Link 평가, Adapter 호출 |
| **DB** | SQLite (better-sqlite3) | 24개 테이블, 동기 I/O |
| **에이전트** | Claude Agent Teams (tmux) | 실제 코드 작업 수행 |

### 데이터 흐름 (한 블록이 완료될 때)

```
1. 팀이 작업 완료 → POST /api/brick/executions/:id/blocks/:block/complete
2. Express → Python 엔진에 전달 (bridge)
3. 엔진: 상태 머신 전이 (running → gate_checking)
4. 엔진: Gate 실행 (command/prompt/metric...)
5. Gate 통과? → Link 평가 → 다음 블록 결정 → 다음 블록 queued
6. Gate 실패? → on_fail 정책 (retry/fail/suspend)
7. 결과 DB 저장 + WebSocket으로 프론트에 실시간 전송
```

### DB 핵심 테이블

| 테이블 | 3축 | 역할 |
|--------|-----|------|
| `brick_block_types` | Brick | 블록 종류 카탈로그 |
| `brick_presets` | Brick | 블록 조합 템플릿 |
| `brick_teams` | Team | 팀 정보 |
| `brick_team_members` | Team | 팀원 |
| `brick_links` | Link | 블록 간 연결 |
| `brick_executions` | Engine | 워크플로우 실행 인스턴스 |
| `brick_gate_results` | Brick | Gate 검증 결과 |
| `brick_approvals` | Brick | CEO 승인 기록 |

## 8. 확장성 — 큰 세계를 만들려면

### 3축 독립 = 확장 가능

각 축이 독립적이니까, 한 축만 확장할 수 있다:

| 확장 | 영향 범위 | 다른 축 영향 |
|------|----------|------------|
| 새 블록타입 추가 (e.g., "Marketing") | Brick만 | Team, Link 변경 없음 |
| 새 Adapter 추가 (e.g., Gemini) | Team만 | Brick, Link 변경 없음 |
| 새 Link 추가 (e.g., "A/B test") | Link만 | Brick, Team 변경 없음 |
| 새 Gate 추가 (e.g., "cost check") | Brick만 | Team, Link 변경 없음 |

### Plugin 확장 (pip install)

Python 엔진은 `entry_points`로 플러그인을 지원한다:

```
pip install brick-adapter-gemini    → 새 Adapter 등록
pip install brick-gate-sonarqube    → 새 Gate 등록
pip install brick-link-ab-test      → 새 Link 등록
```

설치만 하면 엔진이 자동 인식. 코드 수정 없이 확장 가능.

이게 Minecraft 모드와 같은 개념이다. 커뮤니티가 자기만의 블록/도구를 만들어서 공유할 수 있다.

### 마인크래프트 모드처럼

```
Phase 1: 코어 블록만 (stone, wood, dirt)
  → 지금: Plan, Design, Do, Check, Act + Gate 7종 + Link 7종

Phase 2: 레드스톤 (복잡한 회로)
  → 조건부 분기, 루프, 경쟁, 크론 조합

Phase 3: 모드 마켓플레이스
  → 커뮤니티가 만든 블록타입/Gate/Adapter/Link 공유
  → "이 사람이 만든 코드리뷰 Gate가 좋네, 설치하자"
```

### ⚠️ 스텁/미완성 (동작하는 척 하지만 실제로는 안 되는 것)

| 기능 | 문제 | QA 출처 |
|------|------|---------|
| **모든 Gate** | GateExecutor → ConcreteGateExecutor 미연결 | BRK-QA-001 |
| **블록 시작** | adapter_pool 미주입 → StartBlockCommand no-op | BRK-QA-002 |
| **승인 대기** | WAITING_APPROVAL 전환 없음 | BRK-QA-003 |
| **루프백** | check→do, approval→design 미동작 | BRK-QA-004 |
| **resume/cancel** | 엔진 우회, presetId로 잘못된 실행 조회 | BRK-QA-005 |
| **승인/리뷰 인증** | 인증 없이 DB 직접 수정 가능 | BRK-QA-006 |
| **어댑터** | claude_code, codex, webhook 등 스텁 | BRK-QA-012 |
| **프리셋 검증** | 스키마 없이 아무거나 저장 가능 | BRK-QA-013 |

### 🔴 보안 이슈

| 문제 | 위험도 | 설명 |
|------|--------|------|
| Shell Injection | HIGH | Command gate가 문자열 포맷 → subprocess_shell | 
| Auth Bypass | HIGH | 승인/리뷰/override에 인증 없음 |

### 판단

**API 레이어는 94.6% 동작한다.** CRUD, 에러 핸들링, 동시성 가드 다 있다.
**엔진 내부는 기획서와 불일치한다.** Gate/Adapter/승인이 연결 안 돼서, "워크플로우 자동 실행"이라는 핵심 가치가 아직 실현 안 됐다.

Phase 1 버그 수정(BUG-1~7) 끝나면, Phase 2에서 엔진 연동이 핵심이다.
