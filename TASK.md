## T1. 정보공유 글 TOC 카드 스타일 미적용

### 현재 동작
정보공유 글의 "이 글에서 다룰 내용:" 목차(TOC) 섹션이 `<ul>` 태그 없이 bare `<li>` 요소들로 렌더링됨. CSS 규칙 `.post-body > ul:first-of-type`이 TOC가 아닌 다른 `<ul>`(h2 하위 통계 리스트)에 적용되고 있음. TOC 자체는 스타일 없이 plain 텍스트로 표시됨.

### 기대 동작
- TOC 섹션이 카드 스타일(배경, 테두리, 패딩)로 표시됨
- TOC의 각 항목이 리스트로 정렬됨
- 다른 본문 리스트에는 TOC 스타일이 적용되지 않음

### 하지 말 것
- 정보공유 글 본문 내용이나 다른 CSS 규칙 변경
- AI 생성 로직 변경

### 참고
- CSS 파일에 `.post-body > ul:first-of-type` 규칙이 이미 있음
- 문제는 TOC HTML 생성 시 `<ul>` 래핑이 빠져있는 것

## T2. QA 리포팅 챗봇 (채널톡 스타일)

### 현재 동작
QA 이슈를 발견하면 수동으로 정리해서 슬랙이나 문서로 공유해야 함. 체계적인 QA 수집 도구가 없음.

### 기대 동작
- QA 헬프데스크 우하단에 채널톡 스타일 플로팅 챗 버튼
- 버튼 클릭 시 채팅 패널 열림
- 사용자가 글 + 이미지(스크린샷) 첨부 가능
- AI(Haiku)가 입력받은 내용을 QA 항목으로 정리/리스트업
- 제출된 QA 항목들을 목록으로 볼 수 있음
- 관리자만 접근 가능 (수강생 X), 3~4명 사용 규모

### 상세 요구사항
- **플로팅 버튼**: 우하단 고정, 아이콘 + 펄스 애니메이션 등 채널톡 느낌
- **채팅 UI**: 메시지 입력 + 이미지 첨부 버튼 + 전송
- **AI 처리**: Haiku 모델로 사용자 입력을 구조화된 QA 항목으로 변환 (제목, 설명, 심각도, 스크린샷)
- **QA 목록**: 제출된 항목들 리스트 뷰 (날짜, 제목, 상태)
- **저장**: Supabase에 qa_reports 테이블 (또는 적절한 이름)
- **인증**: 관리자 로그인 상태에서만 챗봇 버튼 표시

### 하지 말 것
- 수강생에게 챗봇 노출
- 기존 질문답변(Q&A) 시스템과 혼동/통합
- 복잡한 워크플로우 (단순 수집 + 리스트업만)
- Opus나 Sonnet 같은 큰 모델 사용 (Haiku만)

### 참고
- Anthropic API 키는 이미 환경변수에 있음
- 채널톡 버튼 디자인 참고: 우하단 원형 버튼 + 채팅 패널 슬라이드업

## T3. Q&A 답변 고도화 — CRAG + Adaptive RAG

### 현재 동작
수강생 질문 등록 시 즉시 AI 답변 생성 (after() 트리거). 유사 QA 검색 → 강의 RAG 검색 → Sonnet 답변. 도메인 용어를 이해하지 않고 바로 검색하므로 엉뚱한 결과가 나오고, 웹서치가 없어 강의에 없는 최신 정보를 답변 못 함.

### 기대 동작

**Stage 0: 도메인 인텔리전스 (Sonnet)**
- 질문의 도메인 용어 추출 + 정규화 ("네이버 쇼핑 입점" = 자사몰 상품을 네이버 쇼핑에 노출)
- 질문 의도 파악 ("이 수강생이 진짜 묻고 싶은 것")
- 질문 유형 분류 (강의내용 / 플랫폼현황 / 트러블슈팅 / 비기술)
- 복잡도 판단 (단순 / 중간 / 복잡)
- 단순+비기술 → Stage 1~2 스킵, Sonnet 직접 답변

**Stage 1: 강의 RAG (기존 파이프라인 개선)**
- Stage 0에서 이해한 맥락 기반으로 검색 쿼리 생성
- 기존 벡터 검색 + BM25 키워드 검색 결합 (Hybrid Search)
- 검색 결과 관련성 평가 (Sonnet) → CORRECT / AMBIGUOUS / INCORRECT

**Stage 2: 웹서치 (조건부)**
- 실행 조건: RAG가 AMBIGUOUS 또는 INCORRECT, 또는 플랫폼 현황 질문
- Stage 0의 도메인 이해를 기반으로 검색 쿼리 구성
- Brave Search API 사용
- 결과 관련성 재평가

**Stage 3: 최종 답변 (Sonnet)**
- 원래 질문 + 도메인 컨텍스트 + RAG 결과 + 웹서치 결과 종합
- 강의 내용 기반 + 출처 명시
- Smith 코치 톤 유지

### 핵심 원칙
- **이해가 먼저, 나머지는 그 다음** — 용어를 모르고 검색하면 검색 자체가 엉뚱해짐
- RAG가 1차 소스, 웹서치는 보충
- 모든 단계 Sonnet (Haiku 사용 안 함)

### 하지 말 것
- Haiku 모델 사용 금지 — 전 단계 Sonnet
- 기존 QA 임베딩/승인 플로우 변경
- LangChain/LangGraph 같은 새 프레임워크 도입 — 기존 코드 구조 유지
- 도메인 글로서리를 하드코딩 — Sonnet이 직접 이해

### 참고
- 현재 파이프라인: rag.ts → knowledge.ts → Anthropic API
- 웹서치: Brave Search API (BRAVE_API_KEY 환경변수 필요)
- 아키텍처 문서: mozzi-reports #84
